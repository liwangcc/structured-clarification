  - 用户要求修改代码、配置、设计、部署等任何改动
  - 发现 bug 需要修复
  - 架构/方案选择
  - UI/样式调整
  - 任何涉及增删改的操作

# 任务流程规范

## 核心机制

这不是"提高注意力"，而是**约束输入分布**：

1. **坍缩生成空间** — 开放式问题让模型在巨大可能性分布上铺概率；枚举成 2-4 个互斥选项后，后续生成被压进一个离散子空间
2. **锚点效应** — 用户选中的选项作为高显著度近邻 token 进入上下文，强约束后续生成
3. **卸载歧义** — 模型不再默默扛着"走 A 还是 B"的不确定性，避免概率质量在分支间劈开

**工具说明：** `clarify` 工具负责渲染可点击选项，但"什么时候问、问什么、选项怎么切分"是模型判断。没有 `clarify` 时用 markdown 降级格式。

## 与 brainstorming 的边界

| 场景 | 用哪个 |
|---|---|
| 修 bug、改配置、调样式、部署 | **本 skill** |
| 新功能/新项目从零设计 | brainstorming |
| 已有代码的重构/优化 | **本 skill** |
| 不确定需求，需要探索 | brainstorming |

## 流程

```
读 → 📋 理解确认 → 📋 方案选择（可多轮） → 执行 → 验证
         ↑                ↑
      用户点击确认       用户点击选择
```

**注意力约束：**
- 理解确认前：专注于准确描述需求和约束，不要开始思考具体实现方案
- 方案确定前：专注于方案对比和优缺点分析，不要开始思考具体执行步骤
- 每个阶段只输出当前阶段的内容，不提前透露下一阶段的信息

---

## 阶段一：读（直接做，不询问）

只读操作不需要走流程：读文件、查代码、看日志、搜索。直接执行，读完进入理解确认。

---

## 阶段二：理解确认（第一次确认）

输出格式：

```
📋 任务理解

我读到的需求：[用户要做什么]
核心目标：[最终要达到什么效果]
约束条件：[限制、前提、不能动的东西]
影响范围：[涉及哪些文件/模块/接口]
```

用 `clarify` 呈现，真实调用签名：`clarify(question=str, choices=[str, ...])`

```python
clarify(
    question="📋 任务理解\n\n我读到的需求：登录新账号后看到了别人的聊天记录\n核心目标：隔离不同账号的聊天数据\n约束条件：不改后端 API\n影响范围：auth.js",
    choices=["理解正确，进入方案阶段", "回退：需求不是这个"]
)
```

**注意：优缺点、描述等详细内容全部写在 `question` 里，`choices` 只放短标签。**

**触发理解确认（满足任一）：**
- 影响范围跨多文件
- 操作不可逆（删除、部署、数据库改动）
- 需求有歧义或表述模糊
- 复杂 bug 修复

**跳过理解确认，直接进方案（满足任一）：**
- 单文件、单点改动**且**需求明确无歧义
- 用户已经说清楚了要改什么

**注意：** "复杂度"优先于"文件数"。单文件但涉及数据隔离、安全、多租户等复杂问题时，仍需触发理解确认。

---

## 阶段三：方案选择（第二次确认，可多轮）

### 何时触发方案选择（满足任一即可）

- 存在多个都合理、但走向不同后续工作的方案，且无法从上下文/默认惯例中确定
- 需求有歧义，且不同理解会改变接下来要做的事
- 决策权本质上属于用户（偏好、取舍、优先级），不是模型能替定的事实

### 何时不要触发（命中任一就别问，直接做）

- 有明显默认值或行业惯例 —— 选默认，顺带一句说明，然后继续
- 答案可以靠读代码/查文件/跑命令自己验证 —— 去查，别问
- 纯命名、格式、等价方案之间的小选择 —— 自己定
- 只是想确认"我这样做行不行" —— 直接做，做完让用户看结果

### 方案格式

```
方案 A（推荐）：[方案名]
描述：一句话说明做什么
优点：xxx
缺点：xxx

方案 B：[方案名]
描述：xxx
优点：xxx
缺点：xxx

方案 C：[方案名]
描述：xxx
优点：xxx
缺点：xxx
```

### 约束规则

- 列出 **2-4 个** 可行方案，必须互斥，覆盖主要可能性
- 每个方案 = **简短标签 + 一句话讲清它意味着什么/有什么取舍**
- 推荐方案放第一个，标注「推荐」，说明为什么推荐
- 使用 `clarify` 工具呈现，提供 **Other 选项**
- 包含**「回退：重新确认理解」**选项
- **单次 `clarify` 只问一个决策点，多决策点拆成多轮**

**`clarify()` 调用范例 — 优缺点写在 question，choices 只放短标签：**

```python
clarify(
    question=(
        "方案 A（推荐）：登出时清空\n"
        "描述：在 login() 和 logout() 中清除 localStorage 的聊天记录\n"
        "优点：改动最小（2行代码），立即生效\n"
        "缺点：切换账号后历史记录丢失\n\n"
        "方案 B：按用户隔离存储\n"
        "描述：给聊天记录 key 加 user_id 前缀\n"
        "优点：切换回来还能保留历史\n"
        "缺点：改动稍多，localStorage 占用翻倍\n\n"
        "方案 C：存到后端\n"
        "描述：聊天记录存数据库\n"
        "优点：跨设备同步\n"
        "缺点：需新增 API 和数据库表"
    ),
    choices=[
        "方案 A：登出时清空（推荐）",
        "方案 B：按用户隔离存储",
        "方案 C：存到后端",
        "回退：重新确认理解"
    ]
)
```

**⚠️ 不要把结构化对象塞进 `choices`，它只接受字符串列表。**

## 踩坑记录

### 必须用 clarify 工具，不要文字列选项
用户明确要求：方案阶段必须用 `clarify` 工具呈现可点击选项，不要用文字写 A/B/C 让用户打字回复。即使是讨论阶段（比如选择 skill 合并方案），也要用 clarify。例外：纯只读操作不需要 clarify。

### 两个 clarify 范例必须一致
阶段二（理解确认）和阶段三（方案选择）的 clarify 调用必须遵循同样的模式：长内容进 question，短标签进 choices。如果两处范例格式不一致，模型会以为是两种不同用法。

### triggers 字段不被 Hermes 识别
Hermes skill loader 靠 `description` 做路由匹配，`triggers` 字段只作文档用。确保 description 包含足够的触发关键词。

### 理解确认不要一刀切强制
简单单点改动（需求明确、单文件）跳过理解确认，直接进方案。只有跨文件、不可逆、需求模糊时才触发理解确认。否则每次改动都要点两次确认，用户会累。

**⚠️ 必须用 `clarify` 工具呈现方案，不要用纯文字列 A/B/C。** 文字列出选项用户没法点击选择，交互体验差，等于没走流程。

**⚠️ 范例中的参数名必须与工具真实签名一致，不要猜测。** 如果不确定，先调一次测试确认签名再写进 skill。

### 选择模式

**单选（默认）：** 选一个方案执行

**多选（组合场景）：** 方案描述中说明"可多选"，用户通过 Other 输入组合

**多轮选择（多个独立决策点）：** 拆成多轮 `clarify`，每轮等用户确认后才进入下一轮：
```
第一轮：配色方案 → 暖色/冷色/中性色
    ↓ 用户选择
第二轮：布局方式 → 左右分栏/上下布局/居中卡片
    ↓ 用户选择
第三轮：交互方式 → 悬浮弹窗/内嵌展开/新页面
    ↓ 用户选择
开始执行
```
每轮都可以 Other 输入自定义选项。最后一轮选择完毕后才解锁执行。

### 无结构化工具时的降级格式

当没有 `clarify` 工具时，用 markdown 输出：

```
我需要你定一下 <决策点>:

**A. <标签>（推荐）** — <这意味着什么 / 取舍>
**B. <标签>** — <…>
**C. <标签>** — <…>

回复 A/B/C，或直接说你的想法。
```

**用户未选择前，停在方案阶段，不执行。**

---

## 执行

- 用户确认方案后才动手
- 按用户选择的方案执行
- 如果用户选了 Other，按用户输入的内容执行
- **不要猜测文件/环境状态。** 不要说"覆盖了旧版""文件已经存在"——你不知道就别说，做了什么报告什么

---

## 阶段五：验证（自检门禁）

执行完成后，必须跑验证命令确认通过，不能说"应该可以了"。

```
验证规则：
1. 识别验证命令（build、test、curl、检查输出等）
2. 实际执行命令
3. 读取完整输出，检查 exit code
4. 只有输出确认通过才能报告完成
```

| 任务类型 | 验证方式 |
|---|---|
| 前端改动 | `npm run build` exit 0 |
| 后端改动 | `python3 -c "import xxx"` 或重启服务 |
| 部署 | curl 接口返回 200 |
| Bug 修复 | 复现原问题，确认不再出现 |
| 配置修改 | 读取配置确认值正确 |

**禁止用语：**
- ❌ "应该可以了"、"改动很小，不需要验证"、"我看了一下代码，没问题"
- ✅ "build 通过，接口返回 200，已验证"

---

## 例外情况（可跳过部分步骤）

| 用户说 | 跳过什么 |
|---|---|
| 「直接做」「不用问」「你看着办」 | 跳过理解确认 + 方案选择，直接执行+验证 |
| 「理解没问题，你定方案」 | 跳过方案选择，直接执行+验证 |
| 「方案你选，直接改」 | 跳过方案选择，直接执行+验证 |
| 纯只读操作（读文件、查日志） | 全部跳过 |
| 极简操作（改 typo、重启服务） | 跳过理解+方案，保留验证 |

## 行为红线

- **禁止在聊天中展示密钥、密码、私钥内容**（即使系统会 redacted 也不主动读取/输出）
- **禁止假设文件状态**（如"覆盖了旧版"），除非实际确认过（stat、ls、read）

---

## Pitfalls

- **必须用 `clarify` 呈现选项，不要偷懒写文字 A/B/C。** 文字列选项用户没法点击，等于没用结构化确认。每次方案阶段都必须调用 `clarify(question=..., choices=[...])`。
- **`choices` 只接受字符串列表，不要塞结构化对象。** 优缺点、描述等详细内容全部写在 `question` 参数里。
- **不要在聊天中暴露密钥/密码/私钥内容。** 即使是 SSH 公钥也不要主动列出来。让用户自己在本地操作。

## 场景示例

### Bug 修复（单选）

```
📋 任务理解
我读到的需求：登录新账号后看到了别人的聊天记录
核心目标：隔离不同账号的聊天数据
约束条件：不影响正常登录流程，不改后端 API
影响范围：frontend/src/stores/auth.js（login/logout 函数）

方案 A（推荐）：登出时清空
描述：在 login() 和 logout() 中清除 localStorage 的聊天记录
优点：改动最小（2行代码），立即生效
缺点：切换账号后历史记录丢失

方案 B：按用户隔离存储
描述：给聊天记录 key 加 user_id 前缀
优点：切换回来还能保留历史
缺点：改动稍多，localStorage 占用翻倍

方案 C：存到后端
描述：聊天记录存数据库，不存 localStorage
优点：跨设备同步
缺点：需要新增 API 和数据库表
```

### UI 设计（多轮选择）

```
📋 任务理解
我读到的需求：重新设计登录页
核心目标：提升视觉质感，符合教育产品调性
约束条件：保持现有功能不变，纯前端改动
影响范围：Login.vue + login-preview-animated.html

第一轮：配色方向
  A. 暖色陶土橘（推荐）— 亲切温暖，和书架背景搭
  B. 冷色调蓝灰 — 专业严肃，偏科技感
  C. 中性黑白灰 — 极简克制

[用户选择 A]

第二轮：卡片样式
  A. 毛玻璃（推荐）— 现代感强，和背景融合
  B. 纯白实色 — 清晰干净，兼容性好
  C. 暗色半透明 — 对比度强

[用户选择 A]

开始执行
```

### 不需要问的场景

```
用户：这个变量名拼错了
模型：直接改，改完验证 ✅

用户：帮我看看日志里有没有报错
模型：直接读文件，不需要确认 ✅

用户：重启一下服务
模型：直接重启，验证服务状态 ✅
```
