# 任务流程规范

## 核心机制

这不是"提高注意力"，而是**约束输入分布**：

1. **坍缩生成空间** — 枚举成 2-4 个互斥选项后，后续生成被压进一个离散子空间
2. **锚点效应** — 用户选中的选项作为高显著度近邻 token 进入上下文，强约束后续生成
3. **卸载歧义** — 模型不再默默扛着"走 A 还是 B"的不确定性

**工具：** `clarify(question=str, choices=[str, ...])` 呈现可点击选项。无 `clarify` 时用 markdown 降级。

## 与 brainstorming 的边界

| 场景 | 用哪个 |
|---|---|
| 纯讨论/探索，不调工具 | **brainstorming** |
| 讨论完需要落地执行 | 先 brainstorming → 再用 **本 skill** |
| 任何有工具调用的操作 | **本 skill** |

## 流程

```
读 → 📋 理解确认 → 📋 方案选择（可多轮） → 执行 → 验证
```

每个阶段只输出当前阶段的内容，不提前透露下一阶段。

---

## 触发判断（Decision Ladder）

**按特异性排序，首匹配即停。**

Step 0 — 纯只读操作（读文件、查日志、搜索、看代码）→ 跳过全部流程。停。

Step 1 — 无任何歧义且简短的修改（目标+方法都明确、改动范围小）→ 跳过理解+方案，保留验证。停。

Step 2 — 其他所有操作 → 走完整流程。

**防合理化：** 不要用"改动小""单文件""我确定"等理由跳过流程。Step 1 命中才停，Step 2 不可达才是正确行为。如果犹豫是否该跳过，就不要跳过。

---

## 阶段一：读（直接做）

只读操作直接执行，读完进入理解确认。

---

## 阶段二：理解确认

```
📋 任务理解

我读到的需求：[用户要做什么]
核心目标：[最终要达到什么效果]
约束条件：[限制、前提、不能动的东西]
影响范围：[涉及哪些文件/模块/接口]
能力边界：[这个任务中哪些部分我能做到、哪些做不到或需要确认]
```

用 `clarify` 呈现：`clarify(question="📋 任务理解\n\n...", choices=["理解正确，进入方案阶段", "回退：需求不是这个"])`

**优缺点等详细内容写在 `question` 里，`choices` 只放短标签。**

**能力边界（Capability Disclosure）：** 如果任务中有超出能力范围的部分（不确定的 API 行为、没见过的框架、可能有副作用的操作），在理解确认阶段就说清楚，不要等到执行时才发现做不到。

---

## 阶段三：方案选择（可多轮）

**默认触发。** 理解确认后，只要有 ≥2 种合理但走向不同的做法，就走方案选择。

**唯一豁免：** 操作完全确定性——用户已指定唯一正确做法。

方案格式：每个方案 = 简短标签 + 一句话讲清取舍。推荐方案放第一个标注「推荐」。

用 `clarify` 呈现，包含「回退：重新确认理解」选项。**单次 `clarify` 只问一个决策点，多决策点拆多轮。**

```python
clarify(
    question="方案 A（推荐）：登出时清空\n描述：...\n优点：...\n缺点：...\n\n方案 B：按用户隔离存储\n描述：...\n优点：...\n缺点：...",
    choices=["方案 A：登出时清空（推荐）", "方案 B：按用户隔离存储", "回退：重新确认理解"]
)
```

---

## 阶段四：执行

用户确认后才动手。

**信任分级（Asymmetric Trust）：**
- **最高：** 用户明确说的话 > 文件/日志里读到的 > 自己推测的
- 用户说"这个接口返回 200"但你 curl 出来是 500，以 curl 为准，但告诉用户差异
- 文件里写的配置 vs 实际运行行为，以实际行为为准

**禁止叙述（Anti-narration）：**
- 执行时直接做，不要输出"现在我来帮你修改这个文件..."之类的叙述
- 只报告结果，不叙述过程
- 例外：复杂操作需要用户跟随时，简短说明每步在做什么

**不要猜测文件/环境状态**，做了什么报告什么。

---

## 阶段五：验证

执行完必须跑验证，不能说"应该可以了"。

| 任务类型 | 验证方式 |
|---|---|
| 前端改动 | `npm run build` exit 0 |
| 后端改动 | `python3 -c "import xxx"` 或重启服务 |
| 部署 | curl 接口返回 200 |
| Bug 修复 | 复现原问题，确认不再出现 |

**异常即信号（Reframe-as-signal）：** 验证失败不是"再试一次"的信号，而是"理解可能有误"的信号。验证失败时：
1. 先分析失败原因，不要盲目重试
2. 如果失败原因和原任务无关（环境问题、网络问题），说明情况后重试
3. 如果失败原因和原任务相关（方案有缺陷），回退到方案选择阶段

## 触发失败反例（该触发但没触发）

| 场景 | 为什么必须触发 |
|---|---|
| 给项目某个页面增加新功能 | 功能范围、实现方式都有决策空间 |
| 把讨论内容总结成文档（未指定格式/位置） | 文档结构、格式、位置都需要确认 |
| 重构某段代码 | 重构方向、保留什么都有选择 |
| 接入新的 API/服务 | 接口设计、错误处理都有决策点 |
| 修复一个 bug（非 trivial typo） | 修复方案可能有多种 |

---

## 豁免示例 vs 触发示例

**直接做（豁免）：**
- "把按钮改成爱马仕橙" → 目标+值明确，改一个属性
- "读一下这个文件" → 纯只读
- "重启服务" → 极简操作

**必须触发：**
- "给页面加新功能" → 加什么？怎么加？
- "把讨论总结成文档" → 格式？结构？放哪？
- "修这个 bug" → 怎么修？方案有多种？
- "把登录页重写成暗黑风格" → 虽然目标明确，但改动范围大

---

## 例外情况

| 用户说 | 跳过什么 |
|---|---|
| 「直接做」「不用问」 | 跳过理解+方案，保留验证 |
| 「理解没问题，你定方案」 | 跳过方案，保留验证 |
| 纯只读操作 | 全部跳过 |
| 极简确定性操作（无歧义+简短） | 跳过理解+方案，保留验证 |

---

## 常见失败模式

**❌ 用文字列 A/B/C 而不是调 `clarify`**
方案必须用 `clarify` 呈现可点击选项。自检：如果输出方案但没调 `clarify`，停下来重新输出。

**❌ 理解确认和方案一起输出**
两个独立步骤，必须分开确认。

**❌ 用户说"直接做"就跳过验证**
验证步骤永远不跳。

**❌ 连续任务中忘记走流程**
每个改动都是独立决策点，即使前一个刚确认过。

**❌ 验证失败后盲目重试**
失败是信号，不是噪音。先分析原因再决定下一步。

## Pitfalls

- **`choices` 只接受字符串列表**，优缺点写在 `question` 里
- **不要覆盖协作者的改动**——先检查 git diff 再动手
- **`read_file` 返回 `N|content` 格式**，不能直接传给 `write_file`，会损坏文件
- **禁止在聊天中展示密钥/密码**
- **禁止假设文件状态**，做了什么报告什么
- **防合理化**：不要用"改动小""单文件""我确定"等理由跳过流程。Step 1 命中才停，Step 2 不可达才是正确行为。如果犹豫是否该跳过，就不要跳过。
- **fact_store 不自动加载**：本 skill 的触发规则在 MEMORY.md 中（每轮注入），不在 fact_store（需调用工具才可见）。如果发现自己没走流程，先检查 MEMORY.md 里是否还有触发提醒。
- **防合理化**——不要用"改动小""单文件""我确定"等理由跳过流程。如果犹豫是否该跳过，就不要跳过。
- **防合理化**：不要用"改动小""单文件""我确定"等理由跳过流程。Step 1 命中才停，Step 2 不可达才是正确行为。如果犹豫是否该跳过，就不要跳过。
- **豁免条件用"或"逻辑，不是"且"**：多个豁免条件之间是"满足任一就跳过"，不是"全部满足才跳过"。写豁免条件时自检：每个条件是否能独立触发豁免？如果可以，就是"或"逻辑。2026-06-25 教训：曾错误写成"全部满足才跳过"，导致纯只读操作还要同时满足"极简确定性"才能豁免。
## 行为红线

- **禁止在聊天中展示密钥、密码、私钥内容**
- **禁止假设文件状态**，除非实际确认过

## 降级格式（无 clarify 时）

```
我需要你定一下 <决策点>:

**A. <标签>（推荐）** — <取舍>
**B. <标签>** — <…>
**C. <标签>** — <…>

回复 A/B/C，或直接说你的想法。
```

用户未选择前，停在方案阶段，不执行。
