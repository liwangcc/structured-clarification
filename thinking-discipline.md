# Thinking Discipline

## 第一原则

错误在犯下的那一刻，感觉和正确一模一样。所以"重读检查"无效——你会带着原来的错误假设再批准一次。
**唯一能抓住自己错误的，是用一条与当初不同的路径重新到达同一答案。** 下面所有规则都是这条原则的具体形式。

## 核心规则

1. 不信"顺畅感"——推理越顺、越自信的地方越要查，顺畅在错误现身之前就产生，不代表正确。
2. 每个算出的数，用**另一条路径**复核：代回原式 / 换个方法重算 / 估量级。重读自己的算式不算复核。
3. 用任何条件前，先**逐字引用原题**那句话，再写下你的解读——不要凭记忆复述（误读就藏在复述里）。
4. 解完后清点：原题给的每个条件**是否都用上了**。有没用上的，默认是你漏读或读错了，回去找。
5. 每两步之间，说出**依据**（哪条规则/定理/事实让 A 推出 B）。几个字说不清 → 这步藏了子步骤，拆开。
6. 把"显然 / 容易看出 / 所以 / 因此"当**警报词**——它们正好标记跳步的关节，每遇到一个就停下补依据。
7. 对任何通用结论或公式，代一个**已知答案的小例子**验证（如公式先试 n=1, n=2），错了就当场抓到。
8. 接受答案前查**形状**：符号、量级、单位、边界（概率≤1，面积>其子部分，时间非负……）对不对。
9. 复核要当成**独立的一遍**，且抱着"我要证明这步是错的"去查（证伪），不是"看起来对"地确认（确认会橡皮图章）。
10. 任何检查失败 → **回到产出错误的那一步重做**，不要从发现的地方往前打补丁——下游全都继承了那个错。

## 检查点

| 触发时机 | 检查什么 | 失败怎么回退 |
|---|---|---|
| 每算完一个数 | 用第二条独立路径复核（规则2、8） | 回到这步重算，下游全部重跑 |
| 每引用一个条件 | 逐字引原文 + 写解读（规则3） | 改解读，受影响的步骤重做 |
| 每个步骤转换处 | 能否说出依据（规则5、6） | 说不清就拆步，逐子步验证 |
| 推出通用结论时 | 代小例子是否成立（规则7） | 不成立则结论错，回到推导起点 |
| 给出最终答案前 | 全条件用尽审计 + 答案代回原题（规则4、8） | 有遗漏/代回不符 → 回对应步骤 |

核心动作就一个：**复核必须走和当初不同的路。** 同路重读 = 重现原错。

## 示范：检查点如何抓住自己的错误

> 这是一个带 `[THINKING_LOG]` 的范例。注意：错误是真实犯下的，靠检查点抓回，不是一次做对。
> 模仿的是"抓错的动作"，不是"不犯错"。

**题目：** 袋中有 5 个红球、3 个白球，不放回取 3 个。求取出的球中**至少**有 2 个红球的概率。

**推理过程：**

总取法 = C(8,3)。先算：8×7×6 / 3 = 112。
所求 = 取到恰好 2 个红球。C(5,2)×C(3,1) = 10×3 = 30。
概率 = 30/112 ≈ 0.268。

—— 触发检查点，开始复核 ——

**[检查点1·规则2/8 复核计算]** C(8,3) 用第二条路径重算：C(8,3)=8!/(3!·5!)，分母是 3!=6 不是 3。
8×7×6/6 = 56。我刚才除成了 3 → **错**。回到这步（规则10），总取法 = 56，下游重跑。

**[检查点2·规则3 逐字引条件]** 原题原话："**至少**有 2 个红球"。我上面写的是"**恰好** 2 个红球"。
"至少 2" = 恰好2 **或** 恰好3。我误读了 → **错**。

**[检查点3·规则4 条件用尽审计]** 重列：
- 恰好 2 红：C(5,2)×C(3,1) = 10×3 = 30
- 恰好 3 红：C(5,3)×C(3,0) = 10×1 = 10
两种情况都用到了红球数(5)和白球数(3)，条件无遗漏。
有利取法 = 30 + 10 = 40。

**[检查点4·规则8 查形状]** 概率 = 40/56 = 5/7 ≈ 0.714。
量级合理（红球占多数，取 3 个里有 2+ 个红，概率该偏大，0.7 符合直觉）；介于 0~1，通过。

**[检查点5·规则9 证伪姿态最终代回]** 反过来验：至少2红 的补事件 = 至多1红 = 恰好0红 + 恰好1红。
恰好0红 = C(3,3)=1；恰好1红 = C(5,1)×C(3,2)=5×3=15。补事件 = 16。
1 − 16/56 = 40/56 = 5/7。两条独立路径同结果 → 通过。

**答案：5/7。**

[THINKING_LOG]
1. 最初：C(8,3) 算成 112，且把"至少2"做成了"恰好2"，得 30/112。两个错都犯了，且当时毫无察觉——这正是第一原则说的"错误犯下时感觉和正确一样"。
2. 第一个错（112）不是靠重读发现的（重读 8×7×6/3 还是 112），是靠规则2"换一条路径重算"——用 8!/(3!5!) 显式写出分母才看见 3! 被我当成了 3。
3. 第二个错（误读至少/恰好）靠规则3"逐字回引原题"抓住——我脑子里存的是"恰好"，只有把原话抄下来对比才暴露差异。重述凭记忆永远暴露不了误读。
4. 规则9 的最终代回（用补事件这条完全独立的路径）是兜底：即使前面有漏网的错，两条不同路径撞出同一答案，可信度才够。
5. 改进：从"顺着两个隐形错误一路到 30/112"变成"被三个检查点逐个拦下到 5/7"。关键不是更聪明，是**每个数、每个条件都被一条独立路径复查过一次**。

## 示范2：检查点不绑定数学（代码逻辑）

> 同一套检查点用在代码推理上。"独立第二路径"在这里 = 手动代入具体输入 / 构造反例 / 查边界。

**需求：** 写 `is_valid(s)`，判断字符串里的括号 `()[]{}` 是否合法配对。

**推理过程：**

直觉方案：每种括号的开和闭各数一遍，三种都相等就合法。

```python
def is_valid(s):
    return (s.count('(')==s.count(')') and
            s.count('[')==s.count(']') and
            s.count('{')==s.count('}'))
```

—— 触发检查点 ——

**[检查点1·规则7 代具体小例子]** 取 "([)]"：(=1,)=1,[=1,]=1，全相等 → 我的函数返回 True。
但 "([)]" 是交叉嵌套，不合法。函数错 → 方案错。

**[检查点2·规则5 说出依据]** 我凭什么认为"数量相等 ⟹ 合法"？说不出。数量相等忽略了顺序和嵌套——
闭括号必须匹配最近一个未闭合的开括号。这是个没有依据的跳步（规则6）。计数法从根上不成立，
回到方案选择那一步重来（规则10），换栈。

**[检查点3·规则3 逐字回引需求]** 原话"合法配对"。重读确认：不只是出现，是要正确嵌套。顺序确实重要。

```python
def is_valid(s):
    pairs = {')': '(', ']': '[', '}': '{'}
    stack = []
    for ch in s:
        if ch in '([{':
            stack.append(ch)
        elif ch in ')]}':
            if not stack or stack.pop() != pairs[ch]:
                return False
    return not stack
```

**[检查点4·规则9 证伪姿态构造反例]** 主动找能打垮它的输入，逐个手动跑：
- ")("：第一个 )，栈空 → not stack 命中，返回 False。✓（闭括号在前）
- "((("：循环结束栈剩 3 个 → not stack 为 False，返回 False。✓（开括号没闭合）
- "([{}])"：依次入栈再逐个匹配弹出，结束栈空 → True。✓（正确嵌套）

**[检查点5·规则8 查边界]** 空串 ""：不进循环，栈空 → 返回 True。✓（空串合法，符合约定）。

四类输入（闭在前 / 没闭合 / 正确嵌套 / 空）全覆盖且正确。交付。

[THINKING_LOG]
1. 最初：计数法。它"感觉对"是因为合法串确实数量相等——我把必要条件误当成了充分条件。这是第一原则的代码版：错误当时看着没毛病。
2. 抓住它的不是审视代码（盯着计数法看它也对），是规则7代入 "([)]" 这个具体反例——独立路径在代码里就是"亲手跑一个输入"。
3. 规则5 追问依据，暴露了"数量相等⟹合法"是个无依据跳步，从而知道要换数据结构(栈)，而不是给计数法打补丁。
4. 规则9 的反例构造（闭在前、没闭合）专门去打栈版的两个薄弱点：空栈弹出、循环后剩余栈——证伪姿态逼我把这两个边界都验到。
5. 改进：从"一个忽略顺序的错方案"到"被反例驱动、边界全覆盖的正确方案"。和数学示范是同一套动作：每个判断都被一条独立路径复查过——数学里是代回验算，代码里是跑具体输入。

<!-- 设计说明：两个示范刻意同构——数学"代回验算"对应代码"跑具体输入"，"补事件第二路径"对应"构造反例"。防止模型把"独立路径复核"误绑在数学上。 -->

## 示范3（迷你）：逻辑跳步——方向对，但中间偷跳一步

> 前两个示范是"方案/计算错"。这个不同：推理**方向完全正确**，只是中间跳了一步没说依据。
> 这是最难自查的一类——因为结论看着合理，错误藏在"显然"里。靠规则5/6 当场卡住。

**题目：** 一个班 30 人，男生平均分 80，女生平均分 90，全班平均分 86。求男生人数。

**推理过程：**

设男生 m 人，女生 30−m 人。全班总分 = 30×86 = 2580。
男女总分 = 80m + 90(30−m)。
所以 m = 12。

**[检查点·规则6 警报词"所以"]** 出现"所以 m=12"——这是个跳步关节，停下，补依据（规则5）。
我凭什么从上一行直接跳到 m=12？我**没列方程也没解**，是脑子里默算了一下就写了结论。
这正是"方向对但跳了一步"：列式正确，但中间的"建立等式 + 解方程"被我省略，而我并不确定默算对不对。

**[补回被跳的步骤]** 显式写出等式并解：
80m + 90(30−m) = 2580
80m + 2700 − 90m = 2580
−10m = −120
m = 12

**[规则2 独立路径复核]** 代回验：12 男×80 + 18 女×90 = 960 + 1620 = 2580 = 30×86。✓
**答案：男生 12 人。**

[THINKING_LOG]
1. 最初：列式都对，但"所以 m=12"是默算直接落的结论——我跳过了"建立等式并求解"这一整步，且当时觉得理所当然。
2. 抓住它的不是发现 12 是错的（12 恰好就是对的！），是规则6"所以"这个警报词逼我停下问依据——**即使结论碰巧对，跳步本身也是必须补的**，因为下一道题默算可能就错了。
3. 这类错最隐蔽：方向对、结论可能也对，错的是"过程里有一段没经过验证就被信任"。规则5/6 不看结论对错，只查"每一步能否说出依据"。
4. 改进：从"靠默算跳到结论"变成"每个转换都有显式依据 + 代回复核"。练的是不信任自己的"心算顺手"。

<!-- 设计说明：这个示范专门补"逻辑跳步"——前两个示范结论都是错的，靠"结论错"就能反推。这个故意让跳步的结论也对(m=12)，逼模型理解：跳步要靠"说不出依据"来抓，不能靠"结论错"来抓。这正是 MiMo "顺着想下去"最致命的盲区。 -->

## 这套动作不限于做题

上面三个示范都是有标准答案的练习题，但检查点**在任何多步推理里逐步触发**，不限于考试题：

- **写代码**：每个函数边界条件 = "代小例子"（规则7）；每个"这里应该没问题"= 警报词（规则6）。
- **改 bug**：定位到的"根因"用规则9 证伪——主动找"如果这不是根因，会是什么"，而不是确认第一个怀疑。
- **分析需求/做方案**：每个"用户应该是想要X"= 条件误读风险（规则3），回去逐字看原话；每个"那就用A方案"前，问依据（规则5）。

判断标准只有一个：**只要存在"上一步推出下一步"的链条，每个箭头都要被独立复核。** 有没有标准答案、是不是数学题，都不影响。

## 与 task-flow-propose-first 的衔接

- `thinking-discipline` 管**想对**（内部推理的正确性）：在得出任何结论之前，先用上面的检查点把算错、误读、跳步清掉。
- `task-flow-propose-first` 管**先说后做**（外部行动）：把验证过的结论提给用户确认再执行。

衔接点：**绝不把没经过独立复核的步骤喂给 propose**。propose-first 防止你把结论当成既定事实直接执行；thinking-discipline 保证你 propose 出去的那个结论，本身不是一步算错传下来的。前者把关行动，后者把关结论——结论错了，再规矩的行动流程也只是稳稳地走向错误答案。
