# Structured Clarification Workflow

> 适用于任何 AI 助手。通过两步确认锁定任务方向，减少返工。

## 核心原理

不是"提高注意力"，而是约束输入分布：
1. **坍缩生成空间** — 枚举 2-4 个互斥选项，把开放问题压进离散子空间
2. **锚点效应** — 用户选中的选项作为高显著度 token，强约束后续生成
3. **卸载歧义** — 模型不再扛着多分支的不确定性，输出更聚焦

## 流程

```
读 → 📋 理解确认 → 📋 方案选择（可多轮） → 执行 → 验证
         ↑                ↑
      用户确认           用户选择
```

### 阶段一：读（直接做）

读文件、查代码、看日志。不需要询问用户。

### 阶段二：理解确认

输出任务理解，等用户确认后才进入方案阶段：

```
📋 任务理解

我读到的需求：[用户要做什么]
核心目标：[最终要达到什么效果]
约束条件：[限制、前提、不能动的东西]
影响范围：[涉及哪些文件/模块/接口]
```

等用户回复"理解正确"或纠正后再继续。

**触发条件（满足任一）：**
- 影响范围跨多文件
- 操作不可逆（删除、部署、数据库改动）
- 需求有歧义或表述模糊
- 复杂 bug 修复

**跳过条件（满足任一）：**
- 单文件、单点改动且需求明确
- 用户已经说清楚了要改什么

**注意：** 复杂度优先于文件数。单文件但涉及数据隔离、安全等复杂问题时，仍需触发。

### 阶段三：方案选择

列出 2-4 个方案，每个包含：方案名 + 描述 + 优缺点

```
我需要你定一下 <决策点>:

**A. <标签>（推荐）** — <这意味着什么 / 取舍>
**B. <标签>** — <…>
**C. <标签>** — <…>

回复 A/B/C，或直接说你的想法。
```

规则：
- 推荐项放第一个，说明为什么推荐
- 选项必须互斥，覆盖主要可能性
- 每个选项 = 简短标签 + 一句话讲清取舍
- 用户未选择前不执行
- 单次只问一个决策点，多决策点拆成多轮

**多轮选择：** 多个独立决策点时，拆成多轮，每轮等确认后才进入下一轮。

### 阶段四：执行

用户确认方案后才动手。

### 阶段五：验证

执行完必须跑验证命令，不能说"应该可以了"。

| 任务类型 | 验证方式 |
|---|---|
| 前端改动 | build 通过 |
| 后端改动 | import 通过 / 重启成功 |
| 部署 | 接口返回 200 |
| Bug 修复 | 原问题不再复现 |

## 何时该问

- 多个合理方案，模型无法从上下文中确定
- 需求有歧义，不同理解会导致不同做法
- 决策权属于用户（偏好、取舍、优先级）

## 何时不该问

- 有明显默认值 — 选默认，顺带说明
- 答案可以自己查 — 去查，别问
- 等价小选择 — 自己定
- "我这样做行不行" — 直接做，做完让用户看

## 例外

- 用户说"直接做" → 跳过理解+方案，直接执行+验证
- 纯只读操作 → 全部跳过
- 极简操作（改 typo） → 跳过理解+方案，保留验证
